# 初始化脚手架任务清单

本文档按“一个开发者快速推进”的视角来安排，不追求一步到位，追求尽快出可用版本。

## 1. 第一轮目标

在最短时间内拿到以下可演示能力：

- 可以维护应用实例配置
- 可以从控制台启动一个本地 AI CLI 会话
- 可以看到会话状态变化
- 可以实时看到原始输出
- 可以保存历史记录

## 2. 后端第一轮任务

### 2.1 调整构建基线

当前工程已存在：`D:\Project\ali\260409\mutiAgent\build.gradle`

- `Spring Boot 3.5.x`
- `Java 17`

  - `spring-boot-starter-web`

### 3.4 最小交互流

第一版页面流程建议是：

1. 新建应用实例
2. 在实例页点击“启动会话”
3. 跳转到会话详情
4. 通过 WebSocket 接收原始输出
5. 在列表页看到状态变化
6. 结束后仍可查看消息和原始日志

## 4. Electron 第一轮任务

Electron 不要先做复杂壳能力，只做三件事：

- 启动 Spring Boot 本地服务
- 提供托盘与主窗口
- 提供几个简单 IPC：
  - 选择目录
  - 打开外部终端
  - 打开日志文件目录

## 5. 数据落地方式

### 5.1 SQLite

- 先把数据库文件放到 `%LOCALAPPDATA%\\mutiAgent\\data`
- 由应用启动时自动检测和创建

### 5.2 日志文件

- 应用日志：`logs/app`
- 会话原始日志：`logs/sessions/{yyyyMMdd}/{sessionId}.log`

这样能避免把大量终端输出塞进数据库，导致查询和写入都变慢。

## 7. 风险排序

### 应该先做

- 实例管理
- 会话状态管理
- 原始输出流采集
- SQLite 存储

### 风险高应延后

- 所有 AI 工具统一结构化消息
- TUI 完全页内复刻
- 断线重连后恢复交互现场
- 多平台一致行为

## 8. 一版就该预留的扩展点

虽然第一阶段先做 MVP，但以下抽象要现在就预留：

- `AIAdapter`
- `SessionRuntime`
- `ProcessSupervisor`
- `WebSocket Event Envelope`
- `message.is_structured`

这些一开始不留，后面会非常难补。

## 9. 现在最适合马上执行的动作

如果按开工效率排序，我建议你下一步直接做下面 6 件事：

1. 修改 `build.gradle`
2. 新增 `application.yml`
3. 落地 SQLite 建表 SQL
4. 建立 `controller / application / domain / infrastructure` 骨架
5. 初始化前端 `Vite + Vue 3 + Electron`
6. 打通“创建实例 -> 启动会话 -> WebSocket 输出”的最小链路

## 10. 我给你的明确建议

当前不要先纠结：

- 界面精美程度
- 多 AI 工具一次性全部接入
- 嵌入式交互是否 100% 完整

当前最重要的是先证明这个产品方向成立：  
你能否稳定地把多个本地 AI CLI 会话统一纳入一个控制台。
