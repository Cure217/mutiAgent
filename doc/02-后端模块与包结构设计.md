# 后端模块与包结构设计

## 1. 设计目标

后端不是普通 CRUD 服务，而是本地 AI 会话编排中心。  
所以后端要重点解决四类问题：

- 管理应用实例配置
- 编排会话生命周期
- 托管本地进程 / PTY
- 保存结构化消息与原始终端流

## 3. 分层职责

### `common`

- 放公共枚举、统一异常、响应体、分页对象、时间工具
- 不允许写业务流程

### `config`

- Spring Bean 注册
- WebSocket 配置
- MyBatis 配置
- SQLite 路径、日志目录、线程池配置

### `controller`

- 负责接收前端请求
- 输入输出对象和领域对象隔离
- 一律不直接操作 `ProcessBuilder` 或数据库 Mapper

### `application`

核心编排层，建议至少提供这些服务：

- `InstanceAppService`
- `SessionAppService`
- `HistoryAppService`
- `ConfigAppService`
- `TerminalBridgeService`

典型职责：

- 创建会话
- 校验实例是否启用
- 选择对应 `AIAdapter`
- 调度 `ProcessSupervisor`
- 写入消息和日志
- 推送 WebSocket 事件

### `domain`

放核心规则，不依赖框架：

- `AppInstance`
- `AiSession`
- `MessageRecord`
- `SystemConfig`
- `SessionStatusMachine`

仓储接口也应定义在 `domain` 内，例如：

- `AppInstanceRepository`
- `SessionRepository`
- `MessageRepository`
- `OperationLogRepository`

### `infrastructure`

放所有“和具体技术栈相关”的实现。

#### `persistence`

- `mybatis/mapper`
- `entity`
- `repository`

#### `process`

- `ProcessSupervisor`
- `ProcessRegistry`
- `ProcessHealthChecker`

#### `pty`

- `PtySession`
- `PtyFactory`
- `PtyOutputReader`

#### `adapter`

- `AIAdapter` 接口
- `CodexAdapter`
- `ClaudeAdapter`
- `GenericCliAdapter`
- `AdapterRegistry`

#### `event`

- `SessionEventPublisher`
- `WsEventDispatcher`

#### `storage`

- `SessionRawLogWriter`
- `FileStorageManager`

## 4. 关键接口建议

### `AIAdapter`

```java
public interface AIAdapter {
    String getType();
    LaunchPlan buildLaunchPlan(AppInstance instance, CreateSessionCommand command);
    ParseResult parseOutput(SessionRuntime runtime, String chunk);
    boolean supportsStructuredMessage();
    boolean supportsInteractiveInput();
}
```

设计原因：

- 把“如何启动某个 AI 工具”和“如何解析其输出”封装起来
- 后续接入新工具时，控制影响范围

### `ProcessSupervisor`

```java
public interface ProcessSupervisor {
    SessionRuntime start(SessionLaunchContext context);
    void sendInput(String sessionId, String input);
    void stop(String sessionId, StopMode stopMode);
    Optional<SessionRuntime> find(String sessionId);
}
```

### `SessionRuntime`

建议至少保存：

- `sessionId`
- `appInstanceId`
- `adapterType`
- `interactionMode`
- `status`
- `pid`
- `startedAt`
- `rawLogPath`
- `workdir`
- `envSnapshot`
- `supportsStructured`

## 6. WebSocket 事件建议

为了避免前端和后端后续混乱，建议现在就固定事件协议：

- `session.created`
- `session.status.changed`
- `session.output.raw`
- `session.message.created`
- `session.error`
- `session.closed`

推荐统一事件结构：

```json
{
  "event": "session.message.created",
  "sessionId": "ses_20260409_001",
  "timestamp": "2026-04-09T11:20:30",
  "payload": {}
}
```

## 7. MVP 必须具备的后端能力

- 应用实例 CRUD
- 会话创建、启动、停止
- 查询运行中的会话
- 读取消息列表
- 原始终端输出流转发
- 会话日志文件落盘
- SQLite 持久化
- 基础配置读取

## 8. 第二阶段再加的能力

- PTY 交互输入
- Claude 结构化消息解析
- Codex App Server 适配
- 消息全文检索
- 会话摘要生成
- 异常恢复与自动重连

## 9. 当前工程应优先调整的地方

当前 `build.gradle` 已生成，但与推荐基线有偏差。

建议第一轮就调整：

- `Spring Boot 4.0.5` -> `Spring Boot 3.5.x`
- 引入：
  - `spring-boot-starter-web`
  - `spring-boot-starter-websocket`
  - `spring-boot-starter-validation`
  - `spring-boot-starter-actuator`
  - `org.mybatis.spring.boot:mybatis-spring-boot-starter`
  - `org.xerial:sqlite-jdbc`
  - `com.zaxxer:HikariCP`
  - `org.jetbrains.pty4j:pty4j`

原因：

- 你当前目标是快速做本地工具，不建议一边做业务一边踩 Spring Boot 4 新基线的坑
- 3.5.x 对桌面本地服务已经足够稳定
